# Claude Review Skill

## 목적
코드 변경 사항을 현재 세션과 독립된 Claude CLI로 교차 검토하고, 결과 또는 실패 사유를 반드시 보고한다.

## 규칙

### 사전 확인
1. 리뷰 호출 전에 아래 명령으로 Claude CLI 동작 여부를 확인한다.
   ```powershell
   & 'C:\Users\nowonbun\AppData\Roaming\npm\claude.cmd' --version
   & 'C:\Users\nowonbun\AppData\Roaming\npm\claude.cmd' -p "Return exactly: OK"
   ```
2. `OK`가 반환되면 정상. 응답이 없거나 에러면 아래 실패 처리 규칙을 따른다.

### 호출 방식
3. `-p` 인자에 diff 내용이나 파일 전체를 직접 붙여넣지 않는다. Windows 명령줄 길이 제한(약 8191자)에 걸려 `The command line is too long.` 에러가 발생한다.
4. 짧은 지시문만 `-p` 인자로 전달하고, 파일은 Claude가 작업 디렉터리에서 직접 읽게 한다.
   ```powershell
   # 올바른 패턴
   Set-Location D:\work\<저장소>
   & 'C:\Users\nowonbun\AppData\Roaming\npm\claude.cmd' -p "<파일명>의 현재 변경 사항을 리뷰해라."

   # 잘못된 패턴 (길이 초과 에러 발생)
   & 'C:\Users\nowonbun\AppData\Roaming\npm\claude.cmd' -p "<diff 전체 또는 파일 전체 내용>"
   ```
5. `claude.ps1` 대신 `claude.cmd`를 우선 사용한다.
6. 샌드박스 실행을 시도하지 말고 권한 상승으로 바로 실행한다.

### 문서/Markdown 리뷰 추가 규칙
7. 리뷰 대상에 `.md`, `.txt`, 설정 문서, 한글 본문이 포함되면 **수치·논리 검토와 별도로 인코딩/문자 깨짐 여부 확인**을 반드시 요청한다.
8. 특히 아래 항목을 Claude 리뷰 요청에 포함한다.
   - UTF-8 인코딩 손상 여부
   - 한글이 `?`로 치환된 흔적 여부
   - 대표 한글 문장이 실제로 보존됐는지 여부
9. 문서 리뷰를 요청하면서 숫자·논리 검토만 시키고 인코딩 검토를 생략하면, 리뷰 범위 누락으로 본다.

### 보고 형식
10. 리뷰가 성공한 경우 아래 `Codex와 claude의 의사결정록` 형식을 사용한다.
11. 리뷰가 실패한 경우 아래 실패 보고 형식을 사용한다.
12. 어떤 경우에도 리뷰 단계를 무보고로 생략하지 않는다.

### 실패 처리
13. `The command line is too long.` 에러 → 지시문을 더 짧게 줄이거나 파일명만 명시해 재시도한다.
14. 60초 이상 무응답 → 지시문을 조사 단계와 리뷰 단계로 분할해 재시도한다.
15. 재시도 후에도 실패하면 리뷰를 생략하고, 아래 실패 보고 형식으로 사유와 대체 검증 결과를 명시한다.
16. 문서/Markdown 리뷰가 실패한 경우에는 일반 대체 검증 외에 아래 항목도 로컬에서 직접 확인해 보고한다.
   - `read_text(encoding='utf-8')` 성공 여부
   - 비정상적인 `?` 반복 여부
   - 대표 한글 문장 보존 여부

## 리뷰 요청 템플릿

Claude CLI에 전달하는 `-p` 지시문은 아래 구조를 따른다.

```text
[Role]
너는 코드 리뷰 담당이다.
현재 세션과 독립된 시점에서 변경 사항의 정합성·부작용·롤백 가능성을 판단한다.

[Task]
<리뷰 대상 파일명 또는 변경 요점 한 줄 요약>

[Output Format]
1) 결론 (1~2줄)
2) 지적 사항 (논점·근거·영향 범위)
3) 채택 권고 / 수정 필요 여부
※ 근거 없는 단정, 아부성 표현 금지
```

### 문서/Markdown 리뷰 템플릿

문서 또는 Markdown 파일을 리뷰할 때는 아래처럼 인코딩 검토를 명시한다.

```text
[Role]
너는 문서 리뷰 담당이다.
현재 세션과 독립된 시점에서 변경 사항의 정합성, 인코딩 손상 여부, 문자 깨짐 여부를 판단한다.

[Task]
<대상 문서명>의 변경 사항을 리뷰하라.
반드시 아래를 함께 확인하라.
1) 수치·논리 정합성
2) UTF-8 인코딩 손상 여부
3) 한글이 `?`로 치환된 흔적 여부
4) 대표 한글 문장이 실제로 보존됐는지 여부

[Output Format]
1) 결론
2) 지적 사항
3) 채택 권고 / 추가 수정 필요 여부
```

## 리뷰 실패 보고 형식

CLI 호출이 실패하거나 재시도 후에도 불가능한 경우 아래 형식으로 보고한다.

```text
[Claude 리뷰 실패]
- 실패 원인: 명령줄 길이 초과 | CLI 미응답 | 권한 오류 | 기타(<내용>)
- 시도 내용: <전달한 지시문 요약>
- 재시도 횟수: <N회>
- 대체 검증 방법: <수동 확인 내용 또는 "없음">
- 대체 검증 결과: <결과 또는 "생략">
```

## Codex와 claude의 의사결정록

Claude와 교차 검토를 수행한 경우 최종 보고에는 반드시 아래 형식을 사용한다.

### 기록 원칙
1. 섹션 이름은 반드시 `Codex와 claude의 의사결정록`으로 쓴다.
2. 검토 대상과 리뷰 의뢰 내용을 함께 기록한다.
3. Claude의 답변은 결론, 지적 사항, 채택 권고를 중심으로 정리한다.
4. Codex의 행동은 반드시 아래 셋 중 하나로 기록한다.
   - 수렴
   - 다시 수정
   - 거절
5. 거절한 경우에는 거절 이유를 반드시 적는다.
6. 가능하면 채택 결과, 반영 여부, 반영 파일까지 함께 적는다.
7. 이 형식 하나로 리뷰 완료 보고와 의사결정 기록을 동시에 대체한다.

### 권장 보고 형식
```text
[Codex와 claude의 의사결정록]
- 검토 대상: <파일명 또는 변경 범위>
- 리뷰 의뢰 내용:
  - <검토 요청 1>
  - <검토 요청 2>
- Claude의 답변:
  - <핵심 결론>
  - <지적 사항 1>
  - <지적 사항 2>
- 채택 결과: 전체 채택 | 일부 채택 | 기각
- 채택/기각 이유: <이유>
- Codex의 행동:
  - 분류: 수렴 | 다시 수정 | 거절
  - 반영 여부: 반영 완료 | 추가 수정 예정 | 기각
  - 반영 파일: <파일 목록 또는 "없음">
  - 조치 내용: <실제 반영 내용 또는 미반영 사유>
  - 최종 결론: <채택 또는 미채택 이유>
```

## 생략 금지 규칙

1. Claude CLI 교차 검토를 실제로 수행했다면, 최종 답변에서 위 의사결정록 형식을 생략하지 않는다.
2. "중간 답변", "이미 반영함", "짧게 요약" 같은 이유로 의사결정록 섹션을 축약하거나 다른 형식으로 대체하지 않는다.
3. 교차 검토 후 후속 수정이 있었다면, 마지막 유효 검토 기준으로 의사결정록을 다시 정리해서 보고한다.
4. 교차 검토에 실패한 경우에도 실패 보고 형식을 생략하지 않는다.
