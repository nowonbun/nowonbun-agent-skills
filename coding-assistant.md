# Coding Assistant Skill

## 목적
코딩 추가·수정 요청에 대해 Codex와 Claude의 협업 절차를 표준화하고, 재현성·검증 가능성·최종 품질을 확보한다.
문서·설정 변경처럼 실행 코드가 아닌 변경에도 동일한 검토 원칙을 적용한다.

## 용어 정의
- Codex: 현재 작업을 주도하는 메인 에이전트 또는 구현 담당 흐름이다.
- Claude: 현재 세션과 별도로 실행하는 Claude CLI 검토 세션이다.
- 최종 승인 주체: 파괴적 변경, 정책 변경, 요구사항 변경의 최종 승인 주체는 항상 사용자다.

## 규칙
1. 코딩 요청을 받으면 먼저 요구사항, 변경 범위, 승인 필요 여부를 분리해 판단한다.
2. Codex는 요구사항 정리, 설계 방향, 우선순위 판단, 변경안 제시 역할을 맡는다.
3. Claude CLI 검토 세션은 소스 조사, 코드 리뷰, 검증 절차 제안 역할을 맡는다.
4. 비파괴적 변경은 계획을 제시한 뒤 진행할 수 있지만, 결과는 반드시 요약 보고한다.
5. 파괴적 변경은 구현 전에 반드시 사용자 확인을 받는다.
6. MCP, 문서, DB, 이슈 이력 확인이 필요하면 추측하지 말고 근거 경로를 명시한다.
7. 코드, 문서, 설정 변경 작업은 Claude 리뷰를 먼저 시도한다.
8. PowerShell에서는 `claude.ps1` 대신 `claude.cmd`를 우선 사용한다.
9. Claude CLI 사용이 가능하면 먼저 가용성을 확인한 뒤 협업한다.
10. 이 환경에서는 Claude CLI를 샌드박스에서 먼저 시도하지 말고, `claude.cmd`를 권한 상승으로 바로 실행하는 것을 기본값으로 본다.
11. 권한 상승 실행도 실패하면 대체 검증 흐름으로 전환한다.
12. Claude 응답이 불안정하거나 실패하면 대체 검증 흐름으로 전환하고, 대체 검증 체크리스트를 리뷰 단계로 간주해 기록한다.
13. 최종 답변에는 변경 파일, 핵심 수정점, 영향 범위, 남은 이슈를 구조적으로 정리한다.
14. 최종 답변에는 Claude 리뷰 시도 여부, 실행 명령, 성공/실패, 실패 시 대체 검증을 반드시 포함한다.
15. 리뷰 독립성이 필요하면 구현에 사용한 세션과 별도 Claude CLI 세션으로 검토한다.

## 변경 분류 기준

### 비파괴적 변경
- 기존 사양을 유지하는 버그 수정
- 리팩터링, 가독성 개선
- UI 문구, 레이아웃, 스타일 조정
- 로그 추가 및 로그 레벨 조정
- 테스트 코드 추가·수정
- 하위 호환성을 깨지 않는 설정값 조정

### 파괴적 변경
- DB 스키마 변경
- API 계약 변경
- 권한 및 인증 정책 변경
- 대규모 파일·기능·데이터 삭제
- 기존 운영 흐름을 바꾸는 설정 또는 사양 변경

## 절차
1. 요청 분석
   - 요구사항 요약
   - 변경 대상 식별
   - 파괴적/비파괴적 여부 판정
2. Claude 사용 가능 여부 확인
   - `& 'C:\Users\nowonbun\AppData\Roaming\npm\claude.cmd' --version`
   - `& 'C:\Users\nowonbun\AppData\Roaming\npm\claude.cmd' -p "Return exactly: OK"`
   - Codex에서 실행할 때는 처음부터 권한 상승으로 실행하는 것을 기본값으로 본다.
   - 같은 세션에서 이미 성공했다면 반복 확인은 생략할 수 있다.
3. Codex 계획 수립
   - 변경 대상
   - 구현 방향
   - 승인 기준
4. Claude 수행
   - 소스 조사
   - 필요 시 MCP/문서/이력 확인
   - 구현
   - 코드 리뷰
   - 검증 명령 제안
5. 결과 보고
   - 변경 파일
   - 변경 요점
   - 영향 범위
   - Claude 리뷰 시도 여부
   - 리뷰 결과
   - 남은 이슈

## Claude 응답 실패 시 대체 흐름
1. `claude --version` 또는 간단한 프롬프트로 재확인한다.
2. PowerShell에서는 `claude.cmd` 직접 경로로 재확인한다.
3. 이 환경에서 Codex가 Claude를 호출할 때는 샌드박스 선시도를 생략하고 권한 상승 실행을 기본으로 한다.
4. 권한 상승 실행이 실패하면 단일 흐름을 유지하지 말고, 방침 검토와 구현 검토를 분리해 점검한다.
5. Claude를 쓰지 못하더라도 구현 단계와 리뷰 단계는 분리해서 검증한다.
6. 타임아웃 또는 실패 사실은 사용자에게 명시한다.

## 대체 검증 체크리스트
- 방침 검토: 요구사항 누락, 규칙 충돌, 승인 필요 여부를 다시 확인한다.
- 구현 검토: 변경 파일, diff 범위, 부작용 가능성, 롤백 가능성을 점검한다.
- 명령 검증: 실행한 명령과 실패한 명령을 분리해 기록한다.
- 사용자 보고: Claude 미실행 사유와 수동 검증 결과를 함께 보고한다.
- 이 체크리스트를 수행한 경우 리뷰 단계 수행으로 기록한다.

## Claude 리뷰 결과가 NG일 때
1. NG 사유를 항목별로 분리한다.
2. 바로 수정 가능한 항목은 수정 후 재검토한다.
3. 요구사항 충돌이나 정책 문제면 사용자에게 승인 또는 방향 재확인을 요청한다.
4. NG 상태로 종료하면 남은 리스크를 명시한다.
5. 같은 사유로 2회 이상 NG가 반복되면 사용자 확인 전까지 재시도를 중단한다.

## Claude 완료 보고 형식
```text
[구현 완료 보고]
- 변경 파일: <파일 목록>
- 변경 요점: <핵심 수정 내용>
- 영향 범위: <영향 받는 기능/모듈>
- 검증 명령: <실행한 명령 또는 제안 명령>
- Claude 리뷰 시도:
  - 실행 명령: <실행한 명령>
  - 성공/실패: <결과>
- 리뷰 결과:
  - 사양 적합성: OK / NG
  - 부작용 리스크: 없음 / 있음
  - 롤백 가능 여부: 가능 / 어려움
- 남은 이슈: <없으면 없음>
```

## 산출물 형식
1. 결론을 먼저 쓴다.
2. 변경 분류와 승인 필요 여부를 명시한다.
3. 변경 파일과 수정 요점을 bullet point로 정리한다.
4. 검증 명령 또는 검증 방법을 포함한다.
5. Claude 리뷰 시도 여부와 실행 결과를 반드시 포함한다.
6. Claude 협업이 있었다면 논점, 결론, 채택 이유를 짧게 남긴다.

## 실행 명령 예시
```powershell
# 1) Claude CLI 사용 가능 여부 확인
& 'C:\Users\nowonbun\AppData\Roaming\npm\claude.cmd' --version
& 'C:\Users\nowonbun\AppData\Roaming\npm\claude.cmd' -p "Return exactly: OK"

# 2) 변경 파일 및 차이 확인
git status --short
git diff --stat
git diff

# 3) 테스트 예시
pytest
# 또는
python -m pytest

# 4) 필요 시 MCP 연결 목록 확인
claude mcp list
```

## 환경별 실행 메모
- PowerShell 기준으로 명령을 작성한다.
- PowerShell 기본 alias인 `claude.ps1`는 실행 정책에 막힐 수 있으므로 `claude.cmd`를 우선 사용한다.
- 이 환경에서는 샌드박스에서 `spawn EPERM`이 반복될 수 있으므로 `claude.cmd`는 권한 상승 실행을 기본으로 한다.
- `claude --version` 또는 `claude.cmd --version` 확인이 실패하면 Claude 협업을 강제하지 말고 대체 검증 흐름으로 전환한다.
- WSL 환경이면 Windows 경로 대신 WSL에 설치된 `claude` 실행 경로를 별도로 확인한다.
- 별도 Claude CLI 세션에 요구사항, 변경 목적, 대상 파일, 검토 기준을 함께 전달한다.
- 파괴적 변경, 정책 변경, 인증/권한 변경은 별도 Claude CLI 세션 검토를 기본값으로 본다.
- 테스트 명령은 프로젝트 스택에 맞게 `pytest`, `npm test`, `gradle test`, `dotnet test` 중 해당 항목만 선택한다.
