# 하네스 엔지니어링

## 목적

스킬 문서를 설명용 문서가 아니라 실행 하네스로 설계해, 짧게 압축해도 해석과 실행 결과가 흔들리지 않게 만든다.

## 배경

- 기존 스킬 문서는 `목적`, `규칙`, `산출물 형식` 중심이라 설명에는 유리하지만, 실제 실행 시 우선순위와 완료 기준이 흐려질 수 있다.
- 스킬 압축을 진행할 때 가장 큰 위험은 문장 길이가 아니라 의미 열화다.
- 의미 열화의 구체적 유형은 아래 `의미 보존 원칙`에서 정의한다.

## 핵심 판단

- 스킬 문서는 길이보다 해석 일관성이 중요하다.
- 따라서 스킬 문서는 자연어 설명서보다 하네스 구조가 더 적합하다.
- 하네스 구조는 금지, 의무, 절차, 완료 판정을 분리해 실행 통제력을 높인다.

## 기본 구조

스킬 문서는 기본적으로 아래 H1 구조를 사용한다.

1. `# Must`
2. `# Do NOT`
3. `# Flow`
4. `# Definition of Done`

각 H1의 역할은 아래와 같다.

- `Must`
  - 반드시 지켜야 하는 규칙
  - 적용 범위
  - 단일 원본
  - 형식 규칙
- `Do NOT`
  - 금지 사항
  - 모호성 방지
  - 의미 약화 방지
- `Flow`
  - 작성 순서
  - 검토 순서
  - 압축 검토 절차
- `Definition of Done`
  - 완료 판정 기준
  - 검증 가능 체크리스트

추가로 `Scope`, `Out of Scope`, `Source of Truth`는 독립 H1이 아니라 기본적으로 `Must` 아래에서 다룬다.

## 세부 구조 규칙

- H2는 각 H1 아래의 주제를 나누는 용도로 쓴다.
- 숫자 목록은 순서가 필요한 절차에만 쓴다.
- `-` 목록은 병렬 규칙, 조건, 점검 항목에 쓴다.
- H1 아래에 장문 설명만 두고 실행 규칙을 생략하지 않는다.
- 기본 스킬 문서는 Markdown(`.md`)로 작성한다.
- 구조적 관계, 절차 흐름, 상태 전이, 승인 흐름처럼 텍스트만으로 오해될 수 있는 내용은 PlantUML(`puml`)로 표현한다.
- Markdown 본문은 규칙의 의미를 고정하고, PUML은 흐름과 구조를 시각적으로 고정하는 용도로 쓴다.

## 왜 이 구조를 쓰는가

### 1. 압축 안정성

- `Must`, `Do NOT`, `Flow`, `Definition of Done`은 역할이 겹치지 않는다.
- 그래서 문장을 줄여도 규칙 강도와 절차가 덜 훼손된다.

### 2. 해석 일관성

- 금지와 의무가 섞이면 해석이 흔들린다.
- 완료 기준이 없으면 문서를 읽고도 언제 끝난 것인지 판단하기 어렵다.
- 절차와 결과 형식을 분리하면 실행과 보고를 혼동하지 않는다.

### 3. 검증 가능성

- `Definition of Done`에 체크 가능한 기준을 두면 문서가 설명문이 아니라 테스트 가능한 규약이 된다.

## 반드시 함께 있어야 하는 요소

하네스 구조만으로는 부족하다. 아래 요소를 함께 둬야 한다.

### Scope

- 이 문서가 어디에 적용되는지 적는다.
- 예:
  - 스킬로 배포되는 `SKILL.md`
  - 스킬 원본 문서

### Out of Scope

- 어디에는 적용하지 않는지 적는다.
- 예:
  - `history/` 문서
  - README
  - AGENTS.md
  - 일반 메모 문서

### Source of Truth

- 어떤 규칙의 단일 원본이 무엇인지 적는다.
- 같은 내용을 여러 문서에 복제하지 않기 위한 장치다.

## 문서별 역할 분리

하네스 엔지니어링 관점에서 문서 체계는 아래처럼 나누는 것이 가장 안정적이다.

### global_instructions.md, CLAUDE.md

- 헌법 문서로 본다.
- 전체 스킬 체계의 공통 원칙을 정의한다.
- 응답 스타일, 판단 원칙, 우선순위, 사용자 프로필, 전역 금지 규칙을 정의한다.
- 개별 스킬의 상세 절차를 직접 풀어쓰기보다, 어떤 방향으로 스킬을 적용해야 하는지 상위 원칙을 제공한다.
- 즉, `global_instructions.md`, `CLAUDE.md`는 개별 하네스 실행서가 아니라 **스킬 시스템의 상위 제어 문서**로 본다.

### 각 폴더의 AGENTS.md

- 해당 폴더의 역할 문서로 본다.
- 해당 폴더 범위에서 따를 기본 하네스 구조를 설명한다.
- 어떤 스킬을 우선 적용하는지, 어떤 상황에서 어떤 스킬을 호출하는지 기술한다.
- 필요하면 MCP 사용 여부, MCP 사용 시 확인할 파라미터/반환 형식, 호출 전 주의사항을 함께 적는다.
- 즉, 각 `AGENTS.md`는 **폴더 역할 + 핵심 규칙 + 로컬 실행 맥락에서 스킬과 MCP를 어떻게 돌릴지 안내하는 운영 문서**로 본다.

### 개별 스킬 문서

- 가능한 한 짧은 실행 카드로 본다.
- 하나의 역할 또는 문제 유형에 대한 실행 규칙을 정의한다.
- 핵심은 `언제 쓰는지`와 `간단한 실행 기준`을 빠르게 전달하는 것이다.
- 전역 원칙과 로컬 운영 규칙을 반복하지 않는다.
- 하네스 구조(`Must / Do NOT / Flow / Definition of Done`)를 기본으로 사용하되, 길게 설명하는 문서가 아니라 짧고 강한 규칙 문서로 유지한다.
- 구조적·절차적 표현 규칙은 위 `세부 구조 규칙`을 따른다.

## 권장 문서 구성

가장 권장하는 책임 분리는 아래와 같다.

- `global_instructions.md`, `CLAUDE.md`
  - 헌법
  - 전역 원칙
  - 우선순위
  - 금지 규칙
- `AGENTS.md`
  - 폴더 역할
  - 핵심 규칙
  - 스킬/MCP 운용 방식
- `skills/*.md`
  - 언제 쓰는지
  - 간단한 기준
  - 필요 최소한의 금지와 완료 기준
- `hooks/` 또는 Hook 관련 문서
  - 역할: 문서/규칙 검증을 사람이 놓치기 전에 자동으로 차단하거나 경고한다.
  - 적용 예시:
    - `eslint` 훅(pre-commit, pre-push)으로 포맷/규칙 위반을 사전 차단한다.
    - Python은 `ruff` 훅으로 린트/포맷 검증을 자동화한다.
    - Java는 `Checkstyle`, `SpotBugs`, `PMD`, `Spotless`(또는 `google-java-format`)를 훅 또는 CI 체크로 연결해 병합 전에 검증한다.
    - `GitHub` 훅(예: GitHub Actions, PR 체크)으로 병합 전 규칙 준수 여부를 검증한다.

이 구조의 목적은 중복 설명을 줄이고, 실제 실행 시 필요한 규칙만 빠르게 찾게 하는 것이다.

## 하네스 구성 순서

하네스는 아래 순서로 구성하는 것을 권장한다.

1. `constitution-rule.md`
2. `vocabulary-rule.md`
3. `skill-create-rule.md`
4. `skill-check-rule.md` / `claude-review-runtime.md` / `skill-modify-history.md`
5. `global_instructions.md`, `CLAUDE.md`, `AGENTS.md` 작성
6. 개별 스킬 작성

### 3번 중심 정합성 루프

- 3번(`skill-create-rule.md`)은 초안 작성 후 자체 검증 루프를 먼저 수행한다.
- 자체 검증은 구조 정합성, 강도 표현 일관성, 완료 기준 검증 가능성, 모호어/중복 제거를 포함한다.
- 3번이 안정화되면 1번(`constitution-rule.md`)과 2번(`vocabulary-rule.md`) 정의와 충돌 여부를 다시 점검한다.
- 충돌이 있으면 1/2번 또는 3번을 수정하고, 다시 3번 자체 검증으로 돌아간다.
- 이 왕복 정합성 루프가 종료된 뒤에 다음 단계(4~6번)로 진행한다.

## 의미 보존 원칙

문서를 압축하거나 재작성해도 아래 항목은 유지돼야 한다.

- 누가 수행하는가
- 언제 적용되는가
- 무엇을 해야 하는가
- 의무/권장/금지 수준
- 선후관계
- 예외 조건
- 실패 시 처리

아래와 같은 변경은 의미 열화로 본다.

- `반드시` → `한다`
- `금지한다` → `지양한다`
- 적용 조건 삭제
- 예외 삭제
- 절차 순서 삭제
- 단일 원본 문서명 삭제

## 하네스 엔지니어링 도입 시 주의점

### 1. 형식만 강제하면 안 된다

- H1 네 개만 맞추고 실제 규칙이 비어 있으면 의미가 없다.

### 2. 모든 문서에 기계적으로 적용하면 안 된다

- 스킬 문서에는 적합하지만, history 문서나 일반 설명 문서에는 그대로 적용하지 않는다.

### 3. Definition of Done은 감상문이면 안 된다

- `잘 작성되면 완료` 같은 문장은 무효다.
- 반드시 검증 가능한 항목으로 써야 한다.

## 현재 채택한 방향

- 스킬 작성 규칙 문서는 하네스 구조를 기본 표준으로 본다.
- `Must / Do NOT / Flow / Definition of Done`을 기본 H1으로 사용한다.
- `Scope`, `Out of Scope`, `Source of Truth`는 `Must` 아래에서 다룬다.
- H2는 주제 구분, 숫자와 `-`는 세부 수칙 구분에 사용한다.
- 완료 판정은 `Definition of Done`에서 검증 가능한 체크 항목으로 관리한다.
- 기본 문서는 Markdown으로 작성하고, 구조적·절차적 표현 규칙은 위 `세부 구조 규칙`을 따른다.
- `global_instructions.md`, `CLAUDE.md`는 전체 스킬 구조와 적용 방향을 다루고, 각 폴더의 `AGENTS.md`는 로컬 맥락에서 스킬과 MCP를 어떻게 운용할지 다룬다.
