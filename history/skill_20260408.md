# skill 수정 이력 - 2026-04-08

> 기준 문서: `skill-modify-history.md`

## 목적
`skill-create-rule.md`를 의미 보존 중심 규칙으로 재정의해 이후 스킬 압축 시 의미 열화 위험을 줄인다.

## 사건 요약
기존 `skill-create-rule.md`는 기본 형식 규칙에는 유효했지만, 규칙 강도, 적용 조건, 예외, 선후관계, 단일 원본, 의미 보존 기준을 충분히 강제하지 못했다.

## 문제 원인

### 1. 형식 중심 규칙에 치우침
frontmatter, 제목, 섹션 같은 외형 규칙은 있었지만, 압축 후 의미가 유지되는지 검증하는 기준이 약했다.

### 2. 규칙 강도와 조건 표기가 약함
의무/권장/금지 수준, 적용 조건, 예외, 실패 처리 같은 실행 규칙 요소를 명시적으로 강제하지 않았다.

### 3. 단일 원본 원칙이 부족함
다른 문서와 규칙이 중복 정의될 때 우선순위와 원본 지정 기준이 약했다.

## 영향
- 스킬 문서를 짧게 줄이는 과정에서 의미 열화가 발생할 위험이 있었다.
- 같은 규칙을 문서마다 다르게 재서술해 해석 차이가 생길 수 있었다.
- 절차형 규칙에서 선후관계와 예외가 빠질 가능성이 있었다.

## 이번에 수정한 내용

### 1. skill-create-rule.md
- 목적을 “형식 통일”에서 “의미 보존 가능한 작성 규칙”으로 재정의했다.
- 하네스 구조를 도입해 H1을 `Do NOT`, `Must`, `Flow`, `Definition of Done` 중심으로 재편했다.
- `Scope`, `Out of Scope`, `Source of Truth`를 `Must` 아래에서 강제했다.
- H2는 주제 단위, 숫자와 `-`는 세부 수칙 단위로 쓰도록 규칙을 고쳤다.
- 구조적 관계, 절차 흐름, 상태 전이, 승인 흐름은 PlantUML(`puml`)로 표현한다는 규칙을 추가했다.
- 절차 흐름은 가능하면 PlantUML(`puml`)로 표현하고, 단순 선형 절차만 숫자 목록으로 허용하도록 수정했다.
- `Definition of Done` 아래에 최소 `Verification`, 필요 시 `Monitoring`을 두도록 구조를 강화했다.
- `Monitoring`은 항상 두고, 불필요한 경우에는 생략 대신 불필요 사유를 적도록 수정했다.
- 모호어 충돌을 피하기 위해 `각 규칙은 가능한 한 하나의 행동만 지시한다`를 `각 규칙은 하나의 행동만 지시한다`로 수정했다.
- Authoring, Compression Review 절차에 PlantUML 예시를 추가했다.
- 이후 중복 표현 문제를 줄이기 위해 Authoring, Compression Review의 PlantUML 예시는 제거하고 숫자 목록만 남겼다.
- Verification의 절차 흐름 조건 문구를 `가능하면` 대신 분기·병렬·되돌아감 조건형으로 수정했다.
- 구조적 관계, 상태 전이, 승인 흐름은 PlantUML(`puml`)을 우선 작성하고, 표현 한계가 있으면 텍스트 설명을 추가하도록 규칙을 정교화했다.
- PlantUML 보조 설명 조건을 `충분히 드러나지 않으면`에서 예외 조건·세부 규칙·해석 기준 기준으로 구체화했다.
- PlantUML 보조 설명 조건을 레이블·노트·가드 조건 기준으로 더 구체화해 판단 모호성을 줄였다.
- PlantUML 우선 대상에 비선형 승인 흐름을 명시하고, 선형 승인·작성 절차는 숫자 목록으로 허용해 자기 적용 충돌을 줄였다.
- 같은 문서 안에서 이미 정의한 규칙을 다시 길게 반복하지 말고 참조형 문장으로 치환한다는 규칙을 추가했다.
- 그 기준에 맞춰 `Section Design`, `Done Design`의 중복 설명을 줄이고 `Structure` 참조형으로 정리했다.
- 이후 남은 중복 1건을 제거하기 위해 `Done Design`의 감시 불필요 사유 문구를 삭제하고 `Structure`를 단일 원본으로 유지했다.
- `숫자 목록은 절차에만 쓴다`는 자기 규칙에 맞추기 위해 `Structure`, `Frontmatter`, `Rule Writing`, `Source of Truth`, `Done Design`의 병렬 규칙 목록을 `-` 목록으로 전환했다.
- `선형 승인·작성 절차`처럼 별도 정의되지 않은 용어를 제거하고 `단순 선형 절차`로 통합했다.
- `Rule Writing`의 중복 금지 문구는 제거하고, 중복 금지는 `Do NOT > Duplication`, 치환 방법은 `Rule Writing`에 남기도록 역할을 분리했다.
- `Definition of Done > Verification`은 추상 판정문 대신 관찰 가능한 검증 항목으로 다시 작성했다.
- `스킬 성격상 필요하면 H1을 추가할 수 있다`를 `위 4개 H1만으로 표현하기 어려운 별도 주제가 있으면`으로 바꿔 마지막 모호어를 제거했다.
- `반복이 필요한 경우`를 `Verification처럼 다른 섹션의 규칙을 점검 항목으로 다시 써야 하는 경우`로 구체화해 반복 허용 조건을 닫았다.
- `Section Design`의 `필요할 때`와 `Rule Writing`의 `필요한 최소 표현`을 조건형 문장으로 바꿨다.
- `Verification`에 H1 순서 검증 항목과 `Rule Writing` 준수 검증 항목을 다시 추가했다.
- `Definition of Done > Verification`은 `Structure`, `Rule Writing`, `Do NOT` 등 상위 규칙 충족 여부를 확인하는 참조형 검증으로 줄여 중복을 줄였다.
- `Verification처럼`이라는 예시 의존 표현을 제거하고, 다른 섹션의 규칙을 점검 항목이나 준수 기준으로 다시 쓰는 일반 조건식으로 바꿨다.
- `Structure`의 의무 규칙 다수에 `반드시`를 추가해 의무 강도를 문장 자체로 명시했다.
- `Structure`의 H2 의무 규칙에는 짧은 정의 섹션 예외를 추가했다.
- `Verification`은 `Structure` 재서술을 줄이고 H1 구성·순서·목록 사용 기준과 PlantUML 기준을 참조형으로 축약했다.
- `Verification`에서 남아 있던 `Frontmatter`, `Definition of Done` 구조 재진술 2건도 참조형 문장으로 치환했다.
- H2 예외 규칙의 `짧은` 표현을 제거하고 `단일 항목` 기준만 남겨 모호성을 줄였다.
- `Verification`의 설명문 한 줄은 제거해 실행 규칙 중심으로 정리했다.
- 존재하지 않는 참조 경로(`## Structure의 ## Verification, ## Monitoring`)는 직접 검증 문장으로 바꿨다.
- 규칙 강도, 적용 조건, 예외, 선후관계, 실패 처리 명시 규칙을 유지했다.
- 압축 전후 의미 보존 기준을 하네스 구조 안에서 유지하도록 재정리했다.
- 완료 판정을 검증 가능한 체크 항목으로 강제했다.

## 재발 방지 규칙
아래 항목은 본 사건에 적용한 구체화 규칙이다. history 기록 의무와 생성 시점의 단일 원본은 `skill-modify-history.md`다.

1. 스킬 문서를 수정하거나 새로 만들 때는 먼저 `skill-create-rule.md`의 의미 보존 규칙을 적용한다.
2. 문장을 줄일 때는 길이보다 행동 결과 동일 여부를 먼저 확인한다.
3. 의무/권장/금지, 적용 조건, 예외, 실패 처리가 필요한 규칙은 축약 과정에서 삭제하지 않는다.
4. 다른 문서가 원본인 규칙은 재정의하지 말고 단일 원본을 명시한다.
5. 규칙 문서를 수정하기로 확정한 시점에 같은 날 history 문서를 바로 생성한다.
6. 구조와 절차를 글만으로 설명하기 어려우면 Markdown 본문만으로 끝내지 말고 PlantUML(`puml`)로 함께 고정한다.
7. 스킬 문서에는 최소 검증 구조를 남기고, 운영성이 있는 경우 감시 구조까지 함께 남긴다.
8. 공통 규칙 문서가 비대해지면 문서군 전용 스킬로 분리해 책임을 나눈다.
9. 절차 흐름 표현 규칙과 완료 구조 규칙은 문서 안에서도 자기 적용되게 유지한다.
