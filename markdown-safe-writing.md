# Markdown Safe Writing Skill

## 목적
Markdown 및 일반 문서를 작성·수정할 때 한글이 깨지지 않도록 작성 방법, 저장 방식, 검증 절차를 강제한다.

## 규칙
1. `.md`, `.txt`, 설정 문서 등 텍스트 파일을 새로 만들거나 수정할 때는 항상 UTF-8 저장을 기본값으로 사용한다.
2. Codex/Claude의 Write/Edit 계열 툴을 직접 사용할 때도 저장 후 UTF-8 검증 절차를 생략하지 않는다.
3. PowerShell에서 파일을 쓸 때는 반드시 `Set-Content -Encoding utf8` 또는 `Out-File -Encoding utf8`을 명시한다. BOM 없는 UTF-8이 필요하면 PowerShell 버전별 동작 차이를 확인하고 별도 처리한다.
4. Windows 기본 콘솔(cp949 등) 환경에서 PowerShell 셸 인라인 문자열, heredoc, 파이프 입력으로 한글 본문을 직접 전달해 저장하지 않는다. 한글이 `?`로 치환될 위험이 있으면 ASCII-safe 방식으로 우회한다.
5. 긴 문서를 셸로 써야 할 때는 아래 우선순위를 따른다.
   - 1순위: Python `Path(...).write_text(..., encoding='utf-8')`
   - 2순위: PowerShell `Set-Content -Encoding utf8`
   - 3순위: 이미 검증된 파일 복사 후 부분 수정
6. 셸 입력 단계에서 한글 손실 가능성이 있거나 WSL/bash ↔ PowerShell 파이프 경로를 거칠 때는 유니코드 이스케이프, Base64, 기존 파일 복사 후 부분 수정 같은 ASCII-safe 방법을 사용한다.
7. 문서 저장 후에는 끝내지 말고 반드시 검증한다.
8. 터미널 출력이 깨져 보여도 파일 자체가 정상일 수 있으므로, 표시 문제와 파일 손상을 구분한다.
9. 기존 문서가 이미 깨졌다면 단순 재저장이 아니라 원문을 다시 주입해 복구한다. `?`로 치환된 데이터는 원래 한글로 자동 복원되지 않는다고 가정한다.
10. 같은 규칙을 여러 문서에 중복으로 쓰지 말고, 공통 규칙은 이 스킬이나 기준 문서에 모은다.
11. 문서 수정 작업 보고에는 최소한 아래를 포함한다.
   - 어떤 파일을 수정했는지
   - 어떤 방식으로 저장했는지
   - UTF-8 검증 결과가 어떤지

## 안전한 작성 패턴
1. 새 문서를 길게 작성할 때는 먼저 로컬 변수나 Python 문자열로 내용을 만들고 파일로 저장한다.
2. 문서 일부만 수정할 때는 전체 덮어쓰기보다 필요한 구간만 바꾼다. 단, 인코딩이 이미 깨졌다면 전체 재작성도 허용한다.
3. 템플릿 문서는 반복 사용 가능하도록 예제보다 규칙 중심으로 작성한다.
4. Markdown에서는 제목 계층을 무리하게 깊게 나누지 않는다. H1 1개, 필요한 H2/H3만 사용한다.

## 검증 절차
1. 저장 직후 Python으로 `read_bytes()`와 `read_text(encoding='utf-8')`를 확인한다.
2. 바이트 검증 시 한글 구간이 `0x3F`(`?`)로 치환되지 않았는지 확인한다.
3. 텍스트 검증 시 `read_text(encoding='utf-8')` 결과에서 한글이 정상 문장으로 읽혀야 통과로 본다.
4. BOM 처리 기준을 명시한다.
   - UTF-8 BOM 없음: 기본 허용
   - UTF-8 BOM 있음(`EF BB BF`): 도구 호환성 문제가 없을 때만 허용
   - BOM 여부가 중요하면 문서별로 명시한다.
5. 터미널에서 깨져 보이면 파일 문제인지 콘솔 표시 문제인지 분리해서 판단한다.
6. `Get-Content -Raw` 결과만 보고 인코딩이 정상이라고 판단하지 않는다.

## 검증 명령 예시
```powershell
@'
from pathlib import Path
p = Path(r'D:\work\sample.md')
raw = p.read_bytes()
text = p.read_text(encoding='utf-8')
print(raw[:80])
print(text[:120])
'@ | python -
```

## 검증 실패 시 처리
1. `read_text(encoding='utf-8')` 결과가 깨지면 단순 재저장을 하지 말고 원문을 다시 주입한다.
2. 바이트에 `0x3F`(`?`)가 비정상적으로 반복되면 셸 입력 단계 손상으로 보고 ASCII-safe 방식으로 다시 쓴다.
3. 콘솔 출력만 깨지고 Python UTF-8 읽기가 정상이라면 파일은 정상으로 판단하고 콘솔 표시 문제로 분리한다.

## 금지 사항
1. 인코딩 지정 없이 PowerShell로 파일을 저장하지 않는다.
2. 한글 본문을 셸에 직접 길게 붙여넣고 결과를 검증 없이 끝내지 않는다.
3. 터미널에서 정상으로 보인다는 이유만으로 UTF-8 검증을 생략하지 않는다.
4. 이미 깨진 파일을 원문 확인 없이 그대로 복사·재저장하지 않는다.
